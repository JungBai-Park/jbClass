# 문서정리 / 마무리 (organize-docs)

세션을 마무리하며 프로젝트 문서를 정리한다. 이번 세션에서 코드/동작/관례가 바뀐
내용을 살펴 ① 아래 대상 문서, ② 이번 세션에 **변경한** 소스 파일의 헤더 코멘트에
반영하고, ③ 단계별 계획(`ToDoList.md`)이 있으면 진척을 갱신하고, ④ 검증용 `Temp/` 를
삭제한다. 문서는 **각 문서의 성격에 맞게** 나눠 반영하며, 같은 사실을 여러 문서에 중복
기재하지 않는다 — 한 사실은 가장 알맞은 한 문서에만 둔다.

> 문서/폴더 이름과 레이아웃은 **해당 프로젝트의 규약(예: `CLAUDE.md`)을 우선** 따른다.
> 아래는 그 규약이 따로 없을 때의 기본 레이아웃이다. 단, **문서·소스 주석은 영어로,
> 소스 주석은 ASCII 문자만(바이트값 128 미만) 사용**하는 것을 기본 관례로 한다.

설계명세서는 **별도 문서로 만들지 않는다** (토큰 절약). 설계/구조에 해당하는 내용은
별도 문서 대신 **소스 파일의 헤더 코멘트와 구현 파일 상단 목차**에 기록한다(아래 5번).

대상 (문서는 프로젝트 루트 기준 `Documents/` 아래):

| 대상 | 성격 | 무엇을 적는가 |
|------|------|--------------|
| `Requirements.md` | 요구사항 (사용자가 직접 요청한 사항) | 사용자가 명시적·직접적으로 요청한 사항만. 군더더기 없이 영어로. |
| (소스코드 코멘트) | 명세/결정 ("어떻게 동작하나") | 요청에 따라 결정된 사항·명세(API/함수 시그니처, 동작 규칙, 기본값 등 외부에서 보이는 행동)는 별도 문서가 아니라 **소스 코멘트에만** 기록 |
| `Learned.md` | 빌드/관례/검증 | 빌드·실행 명령, 환경/플랫폼, 파일 인코딩/충돌 회피 관례, 자동/수동 검증 방법, 시행착오로 알게 된 특성. 영어로, 과하지 않게. |
| `ToDoList.md` | 단계별 계획 (있을 때만) | 완료한 단계를 완료로 표시; 완료 단계의 세부는 간략화/생략. 파일이 없으면 건드리지 않는다. |

## 절차

1. **세션 변경사항 수집** — 이번 세션에서 실제로 바뀐 것을 정리한다. 필요하면
   `git status` / `git diff` 와 대화 맥락을 함께 본다. 아직 코드에 반영되지 않은
   "앞으로 할 합의 사항"은 포함하지 않는다 (이미 결정·구현된 것만).
2. **문서별 분류** — 각 변경을 위 표 기준으로 배정한다. 사용자가 직접 요청한 사항이면
   `Requirements.md`. 그 요청에 따라 결정된 사항·명세(외부에서 보이는 행동)와 내부 구조/
   함수 변경은 별도 문서로 적지 않고 **소스 코멘트(공개 인터페이스·구현 파일의 헤더
   코멘트 + 구현 파일 상단 목차)**(5번)에만 둔다. 빌드/검증 방법·시행착오면 `Learned.md`.
3. **외과적 편집** — 해당 문서의 알맞은 섹션을 찾아 그 자리에 끼워 넣는다.
   - 관련 없는 부분의 문장/서식/순서를 건드리지 않는다.
   - 기존 표현·말투·번호 체계를 그대로 따른다.
   - 새 사실이 들어갈 섹션이 없으면 가장 가까운 섹션에 항목으로 추가한다.
4. **중복 점검** — 같은 내용을 다른 문서에도 적지 않았는지 확인한다. 검증 세부는
   `Requirements.md` 에서 한 줄로만 참조하고 본문은 해당 문서에 둔다.
5. **소스 헤더 코멘트 최신화 (변경한 파일만)** — 이번 세션에서 **실제로 수정한** 소스
   파일에 한해, 추가·변경된 API/구조가 그 파일의 헤더 코멘트(공개 인터페이스 파일의 용도·
   요구사항/의존성·사용 예, 구현 파일 상단의 함수/섹션 목차)에 반영됐는지 확인하고 빠진
   것을 채운다. 설계/구조에 해당하는 내용도 여기로 모은다(별도 설계명세서 없음). 손대지
   않은 파일은 점검하지 않는다. (소스 주석은 **영어로, ASCII 문자만**(바이트값 128 미만) 사용한다.)
   - **원칙: 헤더 주석은 코드 수정 시 인라인으로 갱신**한다(프로젝트 규약, 예: `CLAUDE.md`).
     이 단계는 새로 쓰는 자리가 아니라 **누락분만 보완·검증**하는 안전망이다.
6. **ToDoList 진척 갱신** — `Documents/ToDoList.md` 가 있으면 이번 세션에서 완료한
   단계를 완료로 표시하고, 완료 단계의 세부는 토큰 절약을 위해 간략화/생략한다.
   파일이 없으면 생략한다(별도 계획이 없는 정상 상태).
7. **Temp 정리** — 검증용으로 만든 `Temp/` 디렉토리를 삭제한다(핸드오프 `Temp/HandOff.md`
   포함). gitignore 되어 있어 버전관리에는 영향이 없고, 다음에 필요하면 다시 만든다.
8. **요약 보고** — 어떤 변경을 어느 문서 어느 섹션에 넣었는지, 헤더 코멘트/ToDoList/Temp
   정리 결과와 함께 간단히 알린다.

## 주의

- 문서는 영어로 작성하고 인코딩은 기존 파일 관례를 그대로 유지한다. 소스 코드의
  ASCII-only 주석 규칙과는 별개다.
- 설계명세서 문서는 만들지 않는다 — 설계/구조 내용은 소스 헤더 코멘트와 구현 파일 상단 목차로 간다.
- 메모리 시스템(`~/.claude`)에 저장하지 않는다. 프로젝트의 기록은 항상 `Documents/` 문서로 간다.
- 변경이 한 문서에만 해당하면 그 문서만 고친다 (모든 문서를 반드시 고칠 필요 없음).
- 바꿀 게 없으면 그대로 두고 "반영할 변경 없음"이라고 보고한다.
