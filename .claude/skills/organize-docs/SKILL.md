# 문서정리 / 마무리 (organize-docs)

세션을 마무리하며 ConBox 프로젝트를 정리한다. 이번 세션에서 코드/동작/관례가 바뀐
내용을 살펴 ① 아래 평소 대상 문서, ② 이번 세션에 **변경한** 소스 파일의 헤더 코멘트에
반영하고, ③ 검증용 `Temp/` 를 삭제한다. 문서는 **각 문서의 성격에 맞게** 나눠 반영하며,
같은 사실을 여러 문서에 중복 기재하지 않는다 — 한 사실은 가장 알맞은 한 문서에만 둔다.

토큰 효율을 위해 평소 '문서정리/마무리'는 **요구사양서·개발노트 두 문서만** 다룬다.
설계명세서는 매번 동기화하지 않고(코드+`ConBox.cpp` 목차가 더 정확한 출처다),
**마일스톤 완료 시 또는 "설계명세서 정리해줘"라고 명시할 때만** 갱신한다.

평소 대상 문서 (모두 프로젝트 루트 기준 `Documents/` 아래):

| 문서 | 성격 | 무엇을 적는가 |
|------|------|--------------|
| `요구사양서.md` | 요구/명세 (사용자 관점, "무엇을 해야 하나") | API 시그니처, 동작 규칙, 기본값, 키 동작, 표시 규칙 등 외부에서 보이는 행동 |
| `개발노트.md` | 빌드/관례/검증 | 빌드 명령·플랫폼, 파일 인코딩/충돌 회피 관례, 자동/수동 검증 방법 |

마일스톤 완료/명시 요청 시에만 다루는 문서:

| 문서 | 성격 | 무엇을 적는가 |
|------|------|--------------|
| `설계명세서.md` | 설계/구조 ("어떻게 만들었나") | 클래스/데이터 모델, 주요 흐름, 핵심 함수명, 확장 지점, 한계 |

## 절차

1. **세션 변경사항 수집** — 이번 세션에서 실제로 바뀐 것을 정리한다. 필요하면
   `git status` / `git diff` 와 대화 맥락을 함께 본다. 아직 코드에 반영되지 않은
   "앞으로 할 합의 사항"은 포함하지 않는다 (이미 결정·구현된 것만).
2. **문서별 분류** — 각 변경을 위 표 기준으로 한 문서에 배정한다. 외부 동작이면
   요구사양서, 빌드/검증 방법이면 개발노트. (내부 구조/함수만 바뀐 변경은 평소엔
   문서화하지 않고 코드+`ConBox.cpp` 목차에 맡긴다. 마일스톤 정리 때 설계명세서로 모은다.)
3. **외과적 편집** — 해당 문서의 알맞은 섹션을 찾아 그 자리에 끼워 넣는다.
   - 관련 없는 부분의 문장/서식/순서를 건드리지 않는다.
   - 기존 표현·말투·번호 체계를 그대로 따른다.
   - 새 사실이 들어갈 섹션이 없으면 가장 가까운 섹션에 항목으로 추가한다.
4. **중복 점검** — 같은 내용을 다른 문서에도 적지 않았는지 확인한다. 검증 세부는
   요구사양서에서 한 줄로만 참조하고 본문은 해당 문서에 둔다.
5. **소스 헤더 코멘트 최신화 (변경한 파일만)** — 이번 세션에서 **실제로 수정한** 소스
   파일에 한해, 추가·변경된 API/구조가 그 파일의 헤더 코멘트(공개 헤더 `.h` 의 용도·
   요구사항/의존성·사용 예, 구현 `.cpp` 상단의 함수/섹션 목차)에 반영됐는지 확인하고 빠진
   것을 채운다. 손대지 않은 파일은 점검하지 않는다. (Source 의 .h/.cpp 는 UTF-8 BOM 으로 저장한다.)
6. **Temp 정리** — 검증용으로 만든 `Temp/` 디렉토리를 삭제한다. (gitignore 되어 있어
   버전관리에는 영향이 없고, 다음에 필요하면 다시 만든다.)
7. **요약 보고** — 어떤 변경을 어느 문서 어느 섹션에 넣었는지, 헤더 코멘트/Temp 정리
   결과와 함께 간단히 알린다.

## 주의

- 문서 인코딩은 기존 파일 그대로 유지한다 (UTF-8). Source 코드의 BOM 규칙과는 별개다.
- 메모리 시스템(`~/.claude`)에 저장하지 않는다. 이 프로젝트의 기록은 항상 문서/개발노트로 간다.
- 변경이 한 문서에만 해당하면 그 문서만 고친다 (둘 다 반드시 고칠 필요 없음).
- 바꿀 게 없으면 그대로 두고 "반영할 변경 없음"이라고 보고한다.
- 마일스톤(한 단계의 작업)을 완료해 마무리하는 자리라면, 그때는 설계명세서도 함께 갱신한다.

## 참고

CLAUDE.md 의 "요구사양서 정리해줘" 규칙은 요구사양서 한 문서만 다루는 좁은 버전이다.
"문서정리/마무리"는 평소 요구사양서·개발노트(+변경한 소스 헤더)를 보는 버전이고,
설계명세서는 마일스톤 완료나 명시 요청 시에만 포함한다.
