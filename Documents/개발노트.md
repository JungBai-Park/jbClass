# 개발노트

코드만 봐서는 바로 드러나지 않는 빌드/소스 관례와 검증 방법을 모아 둔다.
세션이 바뀌어도 이 내용이 유지되도록 `CLAUDE.md` 에서 import 한다.

## 빌드

VS 프로젝트는 `Project/ConBox.sln` 에 있다.

빌드 명령 (MSBuild 경로와 솔루션 레벨 플랫폼명에 주의):

```
& "C:\Program Files\Microsoft Visual Studio\2022\Community\MSBuild\Current\Bin\MSBuild.exe" "D:\Work\Study\ConBox\Project\ConBox.sln" /t:Build /p:Configuration=Debug /p:Platform=x64
```

- 32비트는 솔루션 레벨에서 `Platform=Win32` 가 아니라 **`x86`** 이다 (vcxproj 레벨은 Win32). 잘못 주면 MSB4126 오류.
- 출력: `Project/Debug.64/ConBox.exe` (구성별로 `.64` / `.32` 디렉터리).

## Source/ 파일 관례 (이식성 + 데모 앱과의 충돌 회피)

- 데모 앱의 CWinApp 파일은 `Project/ConBoxApp.h` / `ConBoxApp.cpp` (클래스 `CConBoxApp`), 컨트롤은 `Source/ConBox.h` / `ConBox.cpp` (클래스 `CConBox`) 다.
- Source 의 .cpp 는 pch 를 안 거치므로 vcxproj 에서 **PrecompiledHeader=NotUsing** 으로 둔다 (`ConBox.cpp`, `Accessories.cpp` 둘 다).
- 전역 include 경로에 `..\Source` 를 넣지 않는다. 데모는 `#include "../Source/ConBox.h"` 로 명시 포함한다.
- Source 의 .h / .cpp 는 반드시 **UTF-8 BOM** 으로 저장한다. BOM 이 없으면 MSVC 가 CP949 로 오인(C4819)하고, 한글 주석 끝 바이트가 줄 연속으로 오작동할 수 있다. 편집 후 PowerShell `UTF8Encoding($true)` 로 다시 저장한다.
- Source 가 pch / targetver 를 거치지 않아 `_WIN32_WINNT not defined ... MAXVER` 안내가 뜨지만 무해하다 (호스트가 정하도록 의도).

## 문자열 API

- 문자열 입력 API (`set_efont`, `set_kfont`, `print`, `on_enter`) 는 `const char*` (UTF-8) 를 받는다. C++ 문자열 리터럴을 그대로 넘기기 위해 `char*` 가 아닌 `const char*` 로 한다.

## 데모 앱 자동 검증

GUI 앱이라 PowerShell 로 실행한 뒤 화면을 캡처해 확인한다.

- 키 입력: `SendInput` (KEYEVENTF_UNICODE, INPUT 구조체는 x64 에서 40바이트) 또는 `SendKeys`.
- 마우스: `mouse_event` + `SetCursorPos`.
- `PostMessage(WM_CHAR)` 는 대화상자 메시지 펌프 때문에 먹지 않으니 입력 큐 경로를 쓴다.
- ConBox 자식 창은 클래스명이 "Afx..." 로 시작한다 (버튼은 "Button").

## 데모 앱 DPI 설정

- 데모 앱은 `CConBoxApp::InitInstance()` 첫 문장에서 `SetProcessDPIAware()` 를 호출해 **System DPI Aware** 로 만든다(비100% 배율에서 글자 선명). 윈도우/DC 생성 전에 호출해야 한다.
- DPI 인식은 프로세스 단위 속성이라 이식 대상 컨트롤(`CConBox`)이 아니라 **호스트 앱**에서 켠다(ConBox 소스 불변). 인식이 켜지면 `make_font` 의 `GetDeviceCaps(LOGPIXELSY)` 가 실제 DPI 를 돌려줘 폰트·cell 이 실제 픽셀에 맞춰진다.
- 선언적 매니페스트(프로젝트 속성 → 매니페스트 도구 → DPI 인식)도 동일 효과지만 vcxproj 메타데이터 명칭이 버전 의존적이고 틀리면 조용히 무시될 수 있어, 확실한 한 줄 API 호출을 쓴다.
- 검증: 디스플레이 배율을 125%/150% 로 바꿔 실행해 글자가 흐릿하지 않은지, 레이아웃/커서/클릭 위치가 어긋나지 않는지 본다.

## 수동 검증이 필요한 항목

- 한글 IME 실시간 조합 과정(조립 애니메이션)은 한글 키보드 레이아웃이 필요해 자동 캡처로 검증하기 어렵다. 직접 한글로 타이핑해 확인한다. (완성형 한글의 입력·2배폭 렌더링 자체는 검증됨)
- 줄 중간 한글 조합 시 뒤 글자 밀림(삽입 동작)과, 조합 중 커서가 속 빈 2px 테두리 사각형으로 보이는지도 IME 가 필요해 직접 한글 조합으로 확인한다.
- IME 한/영 상태에 따른 커서 폭(한글이면 2칸 블록, 영문이면 1칸 블록)도 IME 모드 전환이 필요해 자동 검증이 어렵다. 직접 한/영 키만 눌러 확인한다. 조합 전이라도 즉시 폭이 바뀌어야 정상. (`is_hangul_mode()` 가 `ImmGetConversionStatus` 의 `IME_CMODE_NATIVE` 로 판별, 전환은 `WM_IME_NOTIFY`/`IMN_SETCONVERSIONMODE` 로 즉시 반영)

## 커서 렌더링

- 커서는 칸을 꽉 채우는 **블록**이다. 색은 멤버 `blend_cursor_color()` 가 `cur_bg`:`cur_fg` 를 `cursor_bg_weight`:`cursor_fg_weight`(기본 6:4) 로 섞어 만든다(배경에 가까운 색이라 블록 위에 전경색 글자를 다시 그려도 잘 보인다). 비율은 공개 API `set_cursor_blend(bg_weight, fg_weight)` 로 바꾼다(합으로 정규화). 폭은 `is_hangul_mode()` 면 2칸, 아니면 1칸.
- **깜빡임 타이머는 없다.** 과거의 `CURSOR_TIMER`/`cursor_on`/`set_cursor_blink`/`set_cursor_size` 는 모두 제거했다(블록 정적 표시라 불필요). 커서 이동/갱신은 해당 동작 지점에서 `InvalidateRect` 로만 다시 그린다.
- 그리기는 `OnPaint` 끝부분에서 한다. 본문 글자는 `TA_BASELINE` 정렬로 그리는데, 커서 블록 위에 기존 글자를 다시 그릴 때도 같은 정렬을 잠시 세팅했다가 복원해야 글자 위치가 맞는다.
