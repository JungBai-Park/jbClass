## 1. ConBox

### 1.1 빌드

VS 프로젝트는 `Project/ConBox.sln` 에 있다.

빌드 명령 (MSBuild 경로와 솔루션 레벨 플랫폼명에 주의):

```
& "C:\Program Files\Microsoft Visual Studio\2022\Community\MSBuild\Current\Bin\MSBuild.exe" "D:\Work\Study\ConBox\Project\ConBox.sln" /t:Build /p:Configuration=Debug /p:Platform=x64
```

- 32비트는 솔루션 레벨에서 `Platform=Win32` 가 아니라 **`x86`** 이다 (vcxproj 레벨은 Win32). 잘못 주면 MSB4126 오류.
- 출력: `Project/Debug.64/ConBox.exe` (구성별로 `.64` / `.32` 디렉터리).

### 1.2 Source/ 파일 관례 (이식성 + 데모 앱과의 충돌 회피)

- 데모 앱의 CWinApp 파일은 `Project/ConBoxApp.h` / `ConBoxApp.cpp` (클래스 `CConBoxApp`), 컨트롤은 `Source/ConBox.h` / `ConBox.cpp` (클래스 `CConBox`) 다.
- Source 의 .cpp 는 pch 를 안 거치므로 vcxproj 에서 **PrecompiledHeader=NotUsing** 으로 둔다 (`ConBox.cpp`).
- 전역 include 경로에 `..\Source` 를 넣지 않는다. 데모는 `#include "../Source/ConBox.h"` 로 명시 포함한다.
- Source 의 .h / .cpp 는 반드시 **UTF-8 BOM** 으로 저장한다. BOM 이 없으면 MSVC 가 CP949 로 오인(C4819)하고, 한글 주석 끝 바이트가 줄 연속으로 오작동할 수 있다. 편집 후 PowerShell `UTF8Encoding($true)` 로 다시 저장한다. (이식 시에도 알 수 있도록 이 BOM 규칙은 `ConBox.h` 상단 헤더 코멘트에도 적어 두었다.)
- Source 가 pch / targetver 를 거치지 않아 `_WIN32_WINNT not defined ... MAXVER` 안내가 뜨지만 무해하다 (호스트가 정하도록 의도).
- VS 솔루션 탐색기의 **필터**(`*.vcxproj.filters` 의 가상 폴더)는 `소스 파일` 하나로 통일한다. 과거 `Source` 필터는 제거됐다. 필터는 IDE 표시용일 뿐 디스크 경로/빌드와 무관하니, 새 소스 파일을 프로젝트에 추가할 때 filters 에는 `소스 파일` 필터로 넣는다.

### 1.3 문자열 API

- 문자열 입력 API (`set_efont`, `set_kfont`, `print`, `on_enter`) 는 `const char*` (UTF-8) 를 받는다. C++ 문자열 리터럴을 그대로 넘기기 위해 `char*` 가 아닌 `const char*` 로 한다.

### 1.4 데모 앱 자동 검증

GUI 앱이라 PowerShell 로 실행한 뒤 화면을 캡처해 확인한다.

- 키 입력: `SendInput` (KEYEVENTF_UNICODE, INPUT 구조체는 x64 에서 40바이트) 또는 `SendKeys`.
- 마우스: `mouse_event` + `SetCursorPos`.
- `PostMessage(WM_CHAR)` 는 대화상자 메시지 펌프 때문에 먹지 않으니 입력 큐 경로를 쓴다.
- ConBox 자식 창은 클래스명이 "Afx..." 로 시작한다 (버튼은 "Button").

### 1.5 데모 앱 DPI 설정

- 데모 앱은 `CConBoxApp::InitInstance()` 첫 문장에서 `SetProcessDPIAware()` 를 호출해 **System DPI Aware** 로 만든다(비100% 배율에서 글자 선명). 윈도우/DC 생성 전에 호출해야 한다.
- DPI 인식은 프로세스 단위 속성이라 이식 대상 컨트롤(`CConBox`)이 아니라 **호스트 앱**에서 켠다(ConBox 소스 불변). 인식이 켜지면 `make_font` 의 `GetDeviceCaps(LOGPIXELSY)` 가 실제 DPI 를 돌려줘 폰트·cell 이 실제 픽셀에 맞춰진다.
- 선언적 매니페스트(프로젝트 속성 → 매니페스트 도구 → DPI 인식)도 동일 효과지만 vcxproj 메타데이터 명칭이 버전 의존적이고 틀리면 조용히 무시될 수 있어, 확실한 한 줄 API 호출을 쓴다.
- 검증: 디스플레이 배율을 125%/150% 로 바꿔 실행해 글자가 흐릿하지 않은지, 레이아웃/커서/클릭 위치가 어긋나지 않는지 본다.

### 1.6 폰트 메트릭(cell 크기) 검증

`calc_cell_size` 가 정한 `cell_w`/`cw`/`natw`/`kwid` 같은 값을 빌드 없이 확인하려면 PowerShell + GDI(P/Invoke)로 같은 측정 로직을 재현한다. 다음 함정들 때문에 시행착오가 있었으니 그대로 따른다.

- **DPI 조건을 데모와 맞춘다.** 측정 프로세스에서도 `SetProcessDPIAware()` 를 호출해야 데모(System DPI Aware)와 같은 `GetDeviceCaps(LOGPIXELSY)` 값이 나온다. 안 하면 100% 가상화(96)로 값이 어긋난다.
- **구조체는 C# 쪽에서 다룬다.** PowerShell 에서 `New-Object LOGFONT` 를 만들어 `[ref]` 로 `CreateFontIndirect` 에 넘기면 null 변환 오류로 폰트 생성이 조용히 실패한다(이전 폰트로 측정돼 값이 틀림). 측정 로직 전체를 `Add-Type` 의 C# static 메서드에 넣고 결과 배열만 돌려받는 방식이 안정적이다.
- **PowerShell 5.1 은 BOM 없는 UTF-8 `.ps1` 을 CP949 로 읽어** 한글 리터럴("가")이 깨진다. `.ps1` 을 UTF-8 BOM 으로 저장하거나, 측정 문자를 코드포인트(U+AC00)로 넘긴다.
- **C 의 정수 변환을 그대로 재현한다.** 포인트→픽셀 `-(LONG)(pt*dpi/72+0.5)` 와 `cw=(int)(kwid/(2*ratio)+0.5)` 는 round-half-up(=`[Math]::Floor(x+0.5)`), 높이 비례보정 `(LONG)(...)` 은 0 방향 절사(=`[Math]::Truncate`). PowerShell `[int]` 는 은행가 반올림이라 맞지 않는다.
- 기본 폰트·100% 배율(DPI 96) 실측 예: `natw`(영문 W) 9, `kwid`(영문 높이로 맞춘 '가') 14, `cw=round(14/1.84)=8` → `[emin 6, natw 9]` 범위 안이라 클램프 없이 `cell_w=8`.

### 1.7 데모 폰트 / 글리프 커버리지

- 데모 앱은 영문 폰트로 **Cascadia Mono** 를 쓴다 (`set_efont("Cascadia Mono", 13, "B")`). 초기화면 도표가 둥근 모서리(`╭╮╰╯`, U+256D~2570)와 로고의 사분면 블록(`▘▝▛▜` 등, U+2598~259F)을 쓰는데, **Consolas 에는 이 글리프들이 없어** 그대로 두면 도표/로고가 깨진다. Cascadia Mono(및 Cascadia Code)는 둘 다 보유한다.
- ConBox 의 기본 영문 폰트는 여전히 **Consolas** 다. 폰트 교체는 데모만의 선택이며 ConBox 소스는 불변이다(초기화면 도표 내용과 마찬가지로 데모의 표현물이라 요구사양서가 아니라 데모 코드 `Project/ConBoxDlg.cpp` 가 단일 출처).
- 폰트가 특정 유니코드 글리프를 실제로 가졌는지는 빌드 없이 GDI 로 확인한다: `GetGlyphIndicesW` 에 `GGI_MARK_NONEXISTING_GLYPHS`(0x1) 플래그를 주면 없는 글리프는 `0xFFFF` 를 돌려준다. 실측상 Consolas 는 단선/이중선 박스(`─│┌╔` 등)와 반블록(`█▀▄▌▐`)은 보유하지만 둥근 모서리와 사분면 블록은 전부 없다. (PS5.1 인코딩 함정 때문에 검사 문자는 코드포인트로 넘긴다 — 위 "폰트 메트릭 검증" 항 참조.)

### 1.8 수동 검증이 필요한 항목

- 한글 IME 실시간 조합 과정(조립 애니메이션)은 한글 키보드 레이아웃이 필요해 자동 캡처로 검증하기 어렵다. 직접 한글로 타이핑해 확인한다. (완성형 한글의 입력·2배폭 렌더링 자체는 검증됨. 터미널 모드에서 확정분만 자식에 전달되며, 조합 인라인 렌더·조합 종료 순서 보장은 M5 예정)
- IME 한/영 상태에 따른 커서 폭(한글이면 2칸 블록, 영문이면 1칸 블록)도 IME 모드 전환이 필요해 자동 검증이 어렵다. 직접 한/영 키만 눌러 확인한다. 조합 전이라도 즉시 폭이 바뀌어야 정상. (`is_hangul_mode()` 가 `ImmGetConversionStatus` 의 `IME_CMODE_NATIVE` 로 판별, 전환은 `WM_IME_NOTIFY`/`IMN_SETCONVERSIONMODE` 로 즉시 반영)

### 1.9 커서 렌더링

- 커서는 칸을 꽉 채우는 **블록**이다. 색은 멤버 `blend_cursor_color()` 가 `cur_bg`:`cur_fg` 를 `cursor_bg_weight`:`cursor_fg_weight`(기본 6:4) 로 섞어 만든다(배경에 가까운 색이라 블록 위에 전경색 글자를 다시 그려도 잘 보인다). 비율은 공개 API `set_cursor_blend(bg_weight, fg_weight)` 로 바꾼다(합으로 정규화). 폭은 `is_hangul_mode()` 면 2칸, 아니면 1칸.
- **깜빡임 타이머**(`CURSOR_TIMER`)로 블록을 깜빡인다. `OnTimer` 가 `cursor_on` 을 토글하고 커서 칸만 `InvalidateRect`. 간격은 멤버 `cursor_blink_ms`(기본 `GetCaretBlinkTime`, 0 이면 깜빡임 없이 항상 표시, 공개 API `set_cursor_blink` 로 변경). 타이머는 `open()` 의 윈도우 생성 뒤에 건다. 입력/이동/출력 핸들러는 `bump_cursor()` 로 `cursor_on=true` + 타이머를 재시작해 입력 직후 한 박자 보이게 한다(꺼지는 프레임이 마침 걸리는 것 방지). 자식이 커서를 숨기면(DECTCEM `?25l` → `cursor_visible=false`) 블록을 그리지 않는다. (과거 `set_cursor_size` 는 제거된 상태 그대로다.)
- 그리기는 `OnPaint` 끝부분에서 한다. 본문 글자는 `TA_BASELINE` 정렬로 그리는데, 커서 블록 위에 기존 글자를 다시 그릴 때도 같은 정렬을 잠시 세팅했다가 복원해야 글자 위치가 맞는다.

## 2. ConExe

`Source/ConExe.h` / `ConExe.cpp` (클래스 `CConExe`). 빌드/Source 관례는 §1.1~§1.2 와 동일
(x64 Debug MSBuild, UTF-8 BOM, vcxproj 에 PrecompiledHeader=NotUsing 로 추가, filters 는 `소스 파일`).
vcxproj/filters 에는 ConBox 와 나란히 ConExe.h/.cpp 를 등록해 두었다.

### 2.1 ConPTY 요구·헤더 함정 (Source 가 targetver 를 안 거치는 데서 옴)

- ConPTY 는 **Windows 10 1809(빌드 17763, RS5) 이상**에서만 동작·선언된다.
- **MFC 와 windows.h 포함 순서**: `ConExe.cpp` 는 `ConBox.h`(afxwin.h)를 `ConExe.h`(<windows.h>)
  보다 **먼저** 포함해야 한다. 반대로 두면 afxwin.h 가 `MFC requires use of Winsock2.h` (C1189)
  로 막는다(MFC 는 windows.h 보다 먼저 들어와야 함).
- **버전/속성 매크로**: Source 파일은 호스트의 `targetver.h` 를 거치지 않아 `_WIN32_WINNT` 가
  미정의(MAXVER 로 기본)된다. `ConExe.cpp` 최상단에서 `_WIN32_WINNT`(0x0A00) / `NTDDI_VERSION`
  (0x0A000006, RS5)을 `#ifndef` 로 선지정한다. 그래도 이 SDK 구성에서는 `CreatePseudoConsole`/
  `HPCON` 은 노출되지만 `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE_HANDLE` 매크로가 노출되지 않아,
  없으면 고정값 `0x00020016` 으로 직접 정의한다(SDK 전역에서 동일한 값).

### 2.2 파이프/핸들 수명

- 출력 파이프의 **write 끝(PTY 측)을 자식 생성 후 닫는다**(올바른 정리이며, 입력 read 끝도 함께
  닫는다). 우리가 보관하는 끝은 `in_write`(쓰기)·`out_read`(읽기)뿐.
- **자식 종료 감지는 파이프 EOF 로 안 된다(중요).** ConPTY 에서는 자식이 끝나도 conhost(의사 콘솔)가
  출력 파이프의 write 끝을 계속 쥐고 있어, write 끝을 우리가 닫았어도 `PeekNamedPipe`/`ReadFile` 이
  EOF/broken pipe 를 보지 못한다(영영 닫힘을 못 봄). 그래서 `pump` 끝에서 `WaitForSingleObject(
  proc.hProcess, 0)==WAIT_OBJECT_0` 로 **자식 프로세스 핸들을 직접 폴링**해 종료를 감지하고(남은
  출력을 다 비운 뒤) `handle_child_exit`(→`stop` 후 `exit_cb` 호출)로 정리한다. (M5 에서 `exit`
  안내가 안 뜨던 원인이 이것이었다.)
- 출력 폴링은 메시지 전용 창(`HWND_MESSAGE`)의 16ms 타이머. 호스트에 메시지 펌프(GUI)가 돌아야
  펌프가 동작한다.

### 2.3 검증 (M1·M2·M3·M4·M5)

- 데모(`Project/ConBoxDlg.cpp`)가 `con_exe.attach(&con_box); con_exe.start(...)` 로 자식을 띄운다.
  검증은 §1.4 처럼 PowerShell 로 실행 후 화면 캡처.
  - **M1(출력)**: `cmd /c dir` → 디렉터리 목록이 ConBox 에 표시.
  - **M2(입출력 왕복)**: `cmd`(상호작용) → `SendKeys` 로 `dir`+Enter 전송 → 자식이 받아 실행,
    결과 표시. 입력이 ConBox 로컬 에코가 아니라 자식 에코로 보이는지(로컬 에코 꺼짐) 확인.
  - **M3(VT 파서·셀 그리드·리사이즈)**: 데모 자식을 `powershell -NoLogo` 로 두고 실행 →
    컬러 프롬프트가 색으로 보이고 제어 시퀀스가 글자로 보이지 않음(M1/M2 와 달리 해소). `SendKeys`
    로 `Write-Host ... -ForegroundColor ...`(16색)·`Clear-Host`(화면 클리어 후 좌상단 프롬프트)를
    확인. 창을 좁게 `MoveWindow` 한 뒤 `$Host.UI.RawUI.WindowSize` 가 새 그리드(예 53×16)를
    보고하면 `ResizePseudoConsole` 동기화 성공. (캡처는 `PrintWindow` 플래그 2 로 클라이언트 포함.)
  - **M4(풀스크린)**: powershell 자식 안에서 `& 'C:\Program Files\Git\usr\bin\vim.exe'` 실행
    (Git 의 vim 이 ConPTY 에서 동작) → 대체 화면으로 VIM 로고/틸드/상태줄이 그려지고, 텍스트
    입력·`dd`(줄 삭제 DL, 그 자리에 `~` 채움)·`:q!` 종료 후 주 화면(powershell 프롬프트)·스크롤백
    복원을 캡처로 확인.
  - **M5(claude 구동)**: powershell 자식 안에서 `claude` 실행 → 로고(사분면 블록)·배너·입력창이
    정확히 렌더. 입력 에코·방향키 삽입(`123`+Left Left+`X`=`1X23`, PSReadLine 으로)·종료 콜백
    (`exit` → 노란 종료 안내) 확인. **vt_gtlt 버그 진단**: claude 입력 첫 글자가 (0,0)에 찍히던
    잔상은, ConExe 의 `pump`/`write` 에 임시 I/O 로깅(바이트를 `<ESC>` 등으로 풀어 Temp 에 덤프)을
    넣어 claude 가 보내는 `ESC[<u`(kitty)·`ESC[>4;2m`(XTMODKEYS)를 우리가 표준 `CSI u`/`CSI m` 으로
    오인함을 확인해 고쳤다(`vt_gtlt` 마커). 진단용 로깅은 검증 후 제거한다.
- M1·M2 단계에서는 VT 미해석이라 `←[2J` 등 제어 시퀀스가 글자로 보였으나, M3 의 VT 파서로 해소됐다.

### 2.4 수동 검증이 필요한 항목 (한글 IME)

- 터미널 모드의 한글 조합 종료 순서 보장(요구사양서 3.1)은 M5 에서 구현했다(`finalize_composition`
  이 트리거 키 직전에 `ImmNotifyIME(CPS_COMPLETE)` 로 강제 확정 → 완성분이 트리거보다 먼저 전송).
  다만 실제 합격 판정은 한글 키보드 직접 타이핑으로만 가능하다(자동 캡처 불가). **합격 기준**:
  claude.exe·vim 에서 한글을 조합하다 방향키/Home/End 를 누르면 **조합하던 자리에서 완성된 뒤**
  커서가 이동(`[완성][트리거]` 순서). 옮겨진 칸에서 완성되면 불합격.
- 코드 점검상 두 경로(우리 강제 확정 / IME 가 스스로 먼저 확정) 모두 순서를 보장하고, `OnImeComp`
  의 `n==0` 필터·`return 0` 으로 이중 전송·WM_CHAR 중복을 막는다. `CPS_COMPLETE` 동기성은 Windows
  기본 IME 기준이라 서드파티 IME 에서는 타이밍이 다를 수 있어 실타이핑 확인이 필요하다.
- 조합 인라인 표시는 여전히 시스템 기본 렌더라 ConBox 블록 커서와 겹쳐 보일 수 있다(직접 조합
  렌더는 향후 과제). 한/영 상태에 따른 커서 폭(2칸/1칸)도 한/영 키만 눌러 확인한다(§1.8).
