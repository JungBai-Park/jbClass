# 개발계획서

ConExe 를 도입하며 ConBox 를 **풀스크린 터미널 에뮬레이터**로 키우는 다단계 작업의
로드맵과 진행상황을 담는다. 여러 세션에 걸쳐 진행되므로, 세션이 끊겨도 다음 세션이
**이 문서의 진행상황부터** 이어받도록 `CLAUDE.md` 에서 import 한다.

- 이 문서는 "살아있는 진행 문서"다. 각 마일스톤이 끝나면 그 결과(확정 API·동작·구조·
  검증)를 정식 문서(요구사양서 §3 · 설계명세서 §2 · 개발노트 §2)로 이관하고, 여기에는
  완료 표시만 남긴다.

## 목표

단순 stdin/stdout 리다이렉트가 아니라 **claude.exe / vim / htop 같은 풀스크린 TUI까지
ConBox 안에서 제대로 구동**하는 것이 목표다.

## 확정 결정

| 항목 | 결정 |
|------|------|
| 자식 실행 백엔드 | ConPTY (`CreatePseudoConsole`) |
| ConBox 모델 | 셀 그리드 VT 에뮬레이터로 통합 (기존 append 모델 폐기/흡수) |
| 입력 | 키 raw → VT 인코딩 → PTY stdin |
| 출력 펌프 | 타이머 폴링(`PeekNamedPipe`→`ReadFile`)으로 시작, ConExe 내부 구현이라 후에 워커 스레드로 교체 가능 |
| ConExe ↔ ConBox | ConExe 생성 시 ConBox 지정, 입출력 자동 연결 |
| VT 파서 | 처음부터 필수 (ConPTY 출력 대부분이 VT 시퀀스) |
| 지원 목표 | claude.exe / vim / htop 풀스크린 TUI |

배경 보충:
- ConPTY 는 자식이 진짜 콘솔로 인식하므로 단순 라인 지향 ~ 풀스크린 TUI 가 한 경로로
  동작한다(에뮬레이터가 라인 지향을 포함하는 상위집합이라 이중 모드는 두지 않는다).
- ConPTY 는 stdout/stderr 을 단일 콘솔 출력으로 합친다. "화면에 보여주는" 용도라 적합하며,
  순수 데이터 파이프(출력을 프로그램적으로 가공)와는 다르다.
- 풀스크린은 raw 입력이라 에코·줄편집·커서를 자식이 VT 출력으로 그린다. 그래서 ConBox 의
  기존 입력 줄편집 / `on_enter` / IME 삽입 로직은 셀 그리드 모델로 흡수·축소된다.

## 단계별 로드맵 (각 단계 = 빌드·실행·검증 가능)

매 단계가 "실제 프로그램을 띄워 눈으로 확인"되는 **수직 슬라이스(tracer bullet)** 로
진행한다. 가장 위험한 셀 그리드 재설계(M3)는 ConPTY 왕복(M1~M2)이 검증된 뒤에 들어간다.

### M1 — ConPTY 왕복(출력 단방향)
- 신규 `Source/ConExe.h` / `ConExe.cpp` (클래스 `CConExe`):
  - 입력/출력 파이프 쌍 생성 → `CreatePseudoConsole` 로 PTY.
  - `STARTUPINFOEX` + `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE_HANDLE` 로 자식 `CreateProcess`.
  - 출력 파이프를 UI 타이머(16~30ms)로 `PeekNamedPipe`→`ReadFile` 해 ConBox 에 전달.
- ConBox: 받은 UTF-8 바이트를 **임시로 기존 `print()`** 로 표시(VT 미해석).
- 검증: 데모에서 `cmd /c dir` 출력이 ConBox 에 뜬다. (기존 ConBox 거의 불변 → 위험 낮음)

### M2 — 입력 raw 전송 (왕복 완성)
- ConBox 키 입력(문자/Enter/Backspace/방향키/Ctrl 조합)을 VT 로 인코딩해 `CConExe::write`
  로 PTY 입력 파이프에 쓴다. 로컬 에코는 끈다(자식이 에코).
- 검증: `cmd` 또는 `python -i` 띄워 명령 입력 → 실행·결과 표시. (VT 파서 전이라 일부
  제어가 날것으로 보일 수 있음 — M3 에서 해소)

### M3 — 셀 그리드 화면 버퍼 + 코어 VT 파서 (가장 큰 단계)
- ConBox 내부를 `cols × rows` **셀 그리드 화면 버퍼**(셀 = 글자 + 색/속성)로 전환,
  커서를 `(row, col)` 절대 위치로 관리.
- VT 파서 상태기계: CSI 파싱, 커서 이동(CUP/CUU/CUD/CUF/CUB), 지우기(ED/EL),
  CR/LF/BS/TAB, SGR(기본 16색·굵기/반전 등), 폭 초과 자동 줄넘김.
- PTY 크기 동기화: 그리드 `cols/rows` 변화 → `ResizePseudoConsole`. 창 리사이즈 연동.
- 렌더링: OnPaint 를 셀 그리드 그리기로 교체(기존 셀 렌더·폰트·더블버퍼·커서 자산 재활용).
- 정리 대상: 기존 `log_buf`/`scr_lines`/입력 줄편집/`commit_input`/`on_enter`/IME 삽입 경로.
- 검증: PowerShell/cmd 에서 `cls`·색상 프롬프트·커서 이동 정확, 창 리사이즈 반영.
- 필요하면 **3a 그리드 전환 / 3b VT 코어 / 3c 리사이즈 동기화** 로 더 분할해 검증.

### M4 — 풀스크린 지원
- 대체 화면 버퍼(`?1049h/l`) + 주 화면 스크롤백.
- 스크롤 영역(DECSTBM), 줄/문자 삽입·삭제(IL/DL/ICH/DCH).
- 트루컬러/256색 SGR, 자주 쓰는 모드(커서 표시 `?25`, 자동 줄바꿈 등).
- 검증: `vim` / `htop`(또는 `more`)이 화면을 제대로 그리고 스크롤.

### M5 — claude.exe 구동 + 마감
- 특수키/조합키 VT 인코딩 보강, bracketed paste, (선택) 마우스 SGR 시퀀스.
- 한글 IME: 조합 완성분을 UTF-8 로 PTY stdin 전송(raw 모드 IME 처리).
- 자식 종료 감지 → `on_exit` 콜백, 리사이즈/포커스/정리 마감.
- 검증: **claude.exe** 가 ConBox 안에서 입력/렌더/스크롤/리사이즈 정상 동작.

## 검증 방법 (공통)

- 빌드: 개발노트 §1.1 의 MSBuild 명령(x64 Debug).
- 실행 검증: 데모 앱을 PowerShell 로 띄우고 화면 캡처(개발노트 §1.4). 단계별 자식
  프로그램(`cmd /c dir` → `cmd`/`python -i` → PowerShell `cls`/색상 → `vim`/`htop` →
  `claude.exe`)으로 "그 프로그램이 제대로 보이고 조작되는가"를 합격 기준으로 삼는다.
