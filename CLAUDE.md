# CLAUDE.md



## 프로젝트 개요

1. 타 프로젝트에 쉽게 이식 가능한 ConBox라는 커스텀 윈도우 컨트롤(터미널 컨트롤)에 대한 소스코드를 제작한다.
2. ConBox는 ConPTY로 터미널 기반 응용프로그램을 실행하는 기능을 내장한다.



## 개발환경

- Language: C++ (MFC, unicode)
- OS : Windows 10 64bit



## 작업 방식

- 답변은 항상 격식체( "~합니다.", "~합니까?", "~입니다.", "~하겠습니다.")를 사용한다.
- 답변은 항상 한국어로 답한다.
- 작업과정, 추론과정, 생각하는 내용도 가급적 한국어로 표시한다.
- 특별한 사유가 없는 한, Debug|x64 형태로 빌드하여 시험한다.
- 무언가를 실행 시킬 때 PowerShell을 이용해도 되고 bash를 이용해도 되는 상황에서는,
  운영체제가 Windows 10 64bit인 점을 감안하여 PowerShell을 이용한다.
- 실행을 시켜서 동작을 확인하는 절차는 너무 과하게 하지 않는다.
  - 불확실성이 없는 소스 코드 수정 (당연히 코딩한 대로 동작하게 되는 경우)은 시험을 생략해도 좋다.
  - 용량이 큰 자료(화면 캡춰, 그림 파일 등)을 검토해야 하는 시험은 반드시 
    계획에 시험 내용을 명시하여 사용자한테 알려야 한다. 



## 폴더 구조

- 모든 소스코드는 반드시 ./Source 디렉토리 아래에 생성한다.

```
project-root/
│
├── CLAUDE.md
│
├── Documents/
│    └── Requirements.md, Learned.md 등의 설계 참고 문서
│
├── Project/
│    └── Visual Studio의 프로젝트 폴더 : 비주얼 스튜디오가 생성한 파일들
│
├── Source/
│    └── 소스코드 폴더 : 클로드코드가 생성 및 수정 하는 소스코드들은 여기에
│
├── Reference/
│    └── 단순 참고용 문서 : 클로드가 작업 시 이 폴더의 내용까지 확인해 볼 필요는 없음. (토큰 절약)
│
└── Temp/
     └── 임시 파일 폴더 : 시험/검증용으로 만드는 일회성 파일은 여기에 (gitignore 됨)
```

- 시험·검증을 위해 임시로 만드는 파일(테스트 스크립트, 캡처 이미지, 측정용 ps1, 실험 코드 등)
  -  프로젝트 루트가 아니라 반드시 `Temp/` 디렉토리 아래에 생성한다.
  - `Temp/` 가 없으면 만들어서 쓴다.

## 코딩 규칙 (명명·주석)

- 아래 명명/주석 규칙은 Source/ 밑의 소스코드에 적용하며, Project/ 밑의 소스코드에는 억지로 적용하지 않는다.

- 새 모듈 추가 시 `Source/` 아래에 생성한다. (프로젝트 루트에 .cpp, *.h 파일을 생성하지 않는다.)

- **주석은 충실하게 작성**

  - 주석은 인간이 소스코드를 파악하기 위해 적어 놓은 것이 아니라, 
    AI가 소스코드를 잘 활용할 수 있도록 도움을 주는 목적으로 적는다. 
    (AI에게 도움되지 않는 것은 적지 않는다.)
  - 토큰 절약과 인코딩으로 인한 에러를 방지하기 위해, 
    모든 코멘트는 영어로 적고 ASCII 문자만 사용한다. (바이트 값 128이상 사용 금지)
  - 소스코드를 보면 알 수 있는 내용은 굳이 코멘트로 적지 않는다.
  - 설계 의도, 동작 방식, 사용 시 주의해야 할 점과 같은 내용을 매우 상세히 적는다.

- 파일 헤더 코멘트 (나중에 AI 가 코드를 빠르게 파악하고 토큰을 절약하도록)

  - 각 소스 파일 상단에 그 파일의 용도를 설명하는 헤더 코멘트를 둔다.

  - 별도의 설계명세서 문서를 작성하지 않을 예정이므로, 설계명세서에 반영해야 할 내용들은 헤더코멘트에 기록해 둔다.

    단, 소스코드 전체를 읽지 않아도 되는 상황에서 불필요하게 읽히지 않도록 토큰 효율적인 구조로 정리한다.

  - 공개 인터페이스 헤더(.h)에는 용도·의존성/요구사항·최소 사용 예제를 담아, "사용만" 할 때는 그 헤더만 읽으면 충분하도록 자기완결적으로 작성한다.

  - 구현 파일(.cpp)에는 상단에 함수/섹션 목차를 두어, 전체를 통독하지 않고 필요한 부분만 찾아(Grep/부분 읽기) 접근하도록 한다.

  - 코드를 추가·변경할 때 이 헤더 코멘트와 목차도 그 자리에서(인라인) 함께 최신으로 유지한다. (/organize-docs(마무리) 스킬은 누락분 보완·검증용이지, 헤더 코멘트를 몰아서 쓰는 자리가 아니다.)

-  명명 규칙

  - 파일이름 :
    - PascalCase, 확장자 및 쩜 제외한 이름은 16자 이내 (예: MainCtrl.c)
  - 글로벌 함수 :
    - PascalCase, 대문자로 시작 (예: FunctionNameSample, MyClassName)
    - 가급적 두 단어 이상으로 조합된 이름을 사용한다.
  - 글로벌 변수 :
    - camelCase, 소문자로 시작 (예: functionNameSample, myVariableName)
    - 단, 반드시 대문자가 있어야 하기에, 반드시 두 단어 이상으로 조합된 이름을 사용한다.
  - 멤버변수, 멤버 함수, 로컬 변수 :
    - snake_case, 소문자 + 언더바 (function_name_sample)
    - 단, 로컬변수는 가급적 언더바 없이 짧게 한단어로 표현한다. (정확한 의미 전달이 필요한 경우는 예외로 둔다.)
    - 단순 루프제어의 iteration 변수는 i, n, k 등의 1개 소문자 변수를 사용한다.
    - 멤버 변수를 뜻하는 접두어(m_)는 사용하지 않는다.
  - 매크로 및 enum 상수 :
    - 모두 대문자를 이용하고 여러단어를 조합할 경우 언더바(_)를 사용한다.
  - 단, 개발환경(MFC 등)과의 호환을 위해 불가피한 경우는 예외로 둔다.
