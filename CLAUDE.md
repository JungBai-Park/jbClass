# 프로젝트 개요

아래와 같은 역할을 하는 MFC 기반의 클래스 라이브러리 개발 프로젝트이다.

#### 1. FrameBox 모듈 :

- 대화 상자 컨트롤들의 레이아웃을 동적으로 이동하여 배치하고 이를 소스 코드에 반영한다.
- 대화 상자 컨트롤들의 입력을 while loop 내에서 절차 지향적으로 처리할 수 있는 제어 방식을 제공한다.

#### 2. ConBox 모듈 :

- VT100 호환 터미널 출력 화면 제공
- 차일드 프로세스를 실행하여, stdin/stdout 입출력 연동 (ConPTY 방식)
- 기존 터미널 프로그램들의 한글 입력 불편함 개선

#### 3. TableBox 모듈

- 엑셀과 유사한 느낌의 그리드 컨트롤 제공
- 멤버함수 오버로딩을 이용한 가상 그리드 방식 (화면에 보여지는 데이터만 실시간으로 요청하여 처리)



# 개발 환경

- Language: C++
- OS : Windows 10 64bit, 1809+
- 컴파일러 : Visual Studio 2022, MFC 기반, C++/MFC (Unicode 및 MBCS 호환)
- **MBCS 빌드 호환**:
  모든 MFC/Win32 호출은 명시적 W-접미사 변형(`::CreateFontIndirectW`, `::TextOutW`, `::GetObjectW`, `::CreateEnhMetaFileW`,
  `GetClassNameW`, `CreateWindowExW` 등)을 사용하여 프로젝트의 유니코드/멀티바이트 설정에 무관하게 컴파일 가능해야 한다.
- 텍스트 파일 인코딩 형태 :
  - 소스코드 `(*.c, *.cpp, *.h, *.rc, *.rc2)` :
    무조건 UTF-8 with BOM 형태로 저장한다. (생성한 경우 수정한 경우 모두)
  - 배치 파일 `(*.bat)` :
    무조건 CP949 형태로 저장한다. (생성한 경우 수정한 경우 모두)
  - 그 외 :
    새로 생성할 경우는 UTF-8 with BOM 형태로 인코딩 되어야 한다.
    기존의 파일을 수정할 경우는 원래 인코딩 상태와 동일하게 인코딩되어 저장해야 한다.




# 작업 방식

- 답변은 항상 격식체( "~합니다.", "~합니까?", "~입니다.", "~하겠습니다.")를 사용한다.
- 답변은 항상 한국어로 답한다.
- 작업과정, 추론과정, 생각하는 내용도 가급적 한국어로 표시한다.
- 특별한 사유가 없는 한, Debug|x64 형태로 빌드하여 시험한다.
- 무언가를 실행 시킬 때 PowerShell을 이용해도 되고 bash를 이용해도 되는 상황에서는,
  운영체제가 Windows 10 64bit인 점을 감안하여 PowerShell을 이용한다.
- 주의 : **내가 질문한 내용은 답변만 한다. 답변 내용에 따라 바로 소스 코드를 수정해 놓지 않는다.**
  - 질문한 내용대로 소스 코드를 수정하고자 할 때는 반드시 "진행할까요?" "수정할까요?"를 묻고,
    승인을 받아야 한다.
  - 소스코드 수정에 대해 허락을 받고 승인을 받은 경우와,
    내가 "위 내용을 구현해 줘", "진행시켜 줘"등의 소스코드 수정이 포함된 업무를 요청한 경우만 
    소스코드를 수정 반영한다.

- 실행을 시켜서 동작을 확인하는 절차는 너무 과하게 하지 않는다.
  - 불확실성이 없는 소스 코드 수정 (당연히 코딩한 대로 동작하게 되는 경우)은 시험을 생략해도 좋다.
  - 용량이 큰 자료(화면 캡춰, 그림 파일 등)을 검토해야 하는 시험은 반드시
    계획에 시험 내용을 명시하여 사용자한테 알려야 한다.

# 코딩 규칙

#### 1. 문자열 처리 함수

- `TCHAR` 매크로 또는 컴파일러 옵션에 따라 변동되는 매크로에 의존하지 않고 유니코드 API 함수( `...W`)를 명시적으로 호출한다.
- 유니코드 API 함수를 사용하더라도 `멀티바이트` 코드 용 컴파일러 구성에서 정상적으로 빌드 될 수 있도록 작성한다.
- 내부 변수 및 함수와 연동하는 모든 문자열은 UTF-8로 인코딩된 형태로 사용한다.
- 문자열을 다루기 위한 변수는 char *, const char *, std::string 형식으로 표현한다.
- API와 연동하기 위한 목적 이외에는 다른 형식(wchar *, const char *, CString 등)을 사용하지 않으며,
  다른 형식의 문자열이 사용되더라도 호출 직전/직후에 한해 임시적으로 사용한다.
- ''유니코드 API 함수를 명시적으로 사용한다."는 원칙에 따라,
  컴파일러 옵션에 따라 변동되는 매크로 (`_T()`, `LPCTSTR`, `TCHAR`, `CString` 등)은 사용하지 않는다.

#### 2. 주석

- 주석은 인간이 소스코드를 파악하기 위해 적어 놓은 것이 아니라, AI가 소스코드를 잘 활용할 수 있도록 도움을 주는 목적으로 적는다.
  (AI에게 도움되지 않는 것은 적지 않는다. 예외 : Copyright 문구)
- 토큰 절약과 인코딩으로 인한 에러를 방지하기 위해, 모든 코멘트는 영어로 적는다.
- 공개 인터페이스 헤더(.h)에는 용도·의존성/요구사항·최소 사용 예제를 담아, 
  "사용만" 할 때는 그 헤더만 읽으면 충분하도록 자기완결적으로 작성한다.
- 소스코드 전체를 읽지 않아도 되는 상황에서 불필요하게 읽히지 않도록 토큰 효율적인 구조로 정리한다.
- 바이트 값 128이상 사용 금지하며 ASCII 문자만 사용한다.

  - 단, 이 금지 조항은 오직 주석에만 적용하며, 소스 코드에 한글 문자열이 포함되어야 할 경우에는 한글(UTF-8)로 적는다.
- 소스코드를 보면 알 수 있는 내용은 굳이 코멘트로 적지 않는다.
- 설계 의도, 동작 방식, 사용 시 주의해야 할 점과 같은 내용을 상세히 적는다.
- 중요 :
  PROGRESS.md에 기술된 단계별 계획에 따라 개발이 진행되더라도,
  소스코드 주석에는 Stage 번호 및 단계별 계획에 대한 내용을 기록하지 않는다. (개발 완료된 후 찾아서 지워야 함)
  개발과정이 아니라 개발완료 후 소스코드를 AI가 사용하는 입장에서 필요한 정보만 기록한다.




#### 3. 명명 규칙

- 파일이름 :
  - PascalCase, 확장자 및 쩜 제외한 이름은 16자 이내 (예: MainCtrl.c)
- 글로벌 함수 :
  - PascalCase, 대문자로 시작 (예: FunctionNameSample, MyClassName)
  - 가급적 두 단어 이상으로 조합된 이름을 사용한다.
- 글로벌 변수 :
  - camelCase, 소문자로 시작 (예: functionNameSample, myVariableName)
  - 단, 반드시 대문자가 있어야 하기에, 반드시 두 단어 이상으로 조합된 이름을 사용한다.
- 멤버변수, 멤버 함수, 로컬 변수 :
  - snake_case, 소문자 + 언더바 (function_name_sample)
  - 단, 로컬변수는 가급적 언더바 없이 짧게 한단어로 표현한다. (정확한 의미 전달이 필요한 경우는 예외로 둔다.)
  - 단순 루프제어의 iteration 변수는 i, n, k 등의 1개 소문자 변수를 사용한다.
  - 멤버 변수를 뜻하는 접두어(m_)는 사용하지 않는다.
- 매크로 및 enum 상수 :
  - 모두 대문자를 이용하고 여러단어를 조합할 경우 언더바(_)를 사용한다.
- 단, 개발환경(MFC 등)과의 호환을 위해 불가피한 경우는 예외로 둔다.
